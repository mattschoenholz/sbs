Multi-Location Smartwatch System Design Document

Project Overview

Hardware: Waveshare ESP32-S3 2.06" AMOLED Touch Watch Edition
Locations: House, Van, Boat (OpenPlotter on Pi 5)
Core Requirements:

  • WiFi-based automatic location detection
  • Voice control with audible feedback
  • Location-specific device control
  • Seamless switching between three Home Assistant instances

Component Specification Relevance
Processor ESP32-S3 dual-core 240MHz Sufficient for LVGL + Voice + WiFi
RAM 8MB PSRAM Handles multiple LVGL pages + audio buffers

2. Architecture Decisions

Decision 2.1: Communication Protocol

Option Pros Cons Recommendation
A. Native API Fast, reliable, encryption Cannot switch endpoints dynamically Eliminated
B. MQTT Dynamic broker switching, lightweight, multiple subscribers Requires broker infrastructure Primary
C. HTTP REST Direct control, no broker needed Higher latency, more code Secondary/Fallback
D. Hybrid Best of both worlds More complex configuration Final Choice
Decision: Use MQTT as primary for state publishing and HTTP REST for direct commands when immediate response is needed.

Decision 2.2: MQTT Broker Strategy

Option Architecture Pros Cons Best For
A. Local Brokers Each location runs Mosquitto Low latency, works offline No cross-location visibility, complex bridging Isolated systems
B. Cloud Broker Single broker (HiveMQ/EMQX/CloudMQTT) Simple, global access, easy failover Requires internet, latency, subscription costs Always-connected scenarios
C. Hybrid Cloud Local brokers + cloud bridge Offline capability + remote access Complex setup, more maintenance Your use case
D. VPN Mesh Local brokers + VPN tunneling Secure, private, no cloud dependency Requires VPN infrastructure, battery drain Security-focused
Decision: Option C (Hybrid Cloud) - Local Mosquitto at each location bridged to a cloud broker (or VPS) for redundancy and remote access.

Decision 2.3: Voice Pipeline Architecture

Component Location Rationale
Wake Word Watch (local) Privacy, instant response, no latency
STT (Whisper) Local HA instance Works offline, no data costs, fast
NLP/Intent Local HA instance Context-aware, custom sentences
TTS (Piper) Local HA instance Offline, customizable voices
Audio Output Watch speaker Local feedback
Alternative: Cloud voice services (Alexa, Google) - Rejected due to internet dependency on boat/van.

Decision 2.4: Boat Integration Method

Option Setup Pros Cons Recommendation
A. HA on OpenPlotter Pi Docker container alongside Signal K Unified ecosystem, voice control, automation Resource usage, single point of failure Recommended
B. Direct Signal K REST Watch talks directly to Signal K No HA needed, simpler No voice pipeline, limited automation Backup option
C. MQTT Only Signal K MQTT plugin, no HA Lightweight No voice, limited processing Not recommended
Decision: Option A - Run Home Assistant in Docker on the Pi 5. Use Signal K Home Assistant integration for seamless data flow.

3.2 Location Detection Logic

Detection Method Implementation Reliability Notes
SSID Match wifi_info.ssid sensor ⭐⭐⭐⭐⭐ Primary method
BSSID Match Specific MAC address ⭐⭐⭐⭐⭐ Confirms specific AP
IP Range Subnet detection ⭐⭐⭐⭐ Secondary confirmation
GPS (if added) External GPS module ⭐⭐⭐⭐⭐ Ultimate fallback
BLE Beacon Fixed BLE beacons ⭐⭐⭐ Indoor confirmation

Implementation Code

text_sensor:
  - platform: wifi_info
    ssid:
      id: current_ssid
      on_value:
        then:
          - lambda: |-
              std::string ssid = x;
              int new_location = 0;
              
              if (ssid == "HouseNetwork") {
                new_location = 1;
                id(mqtt_broker) = "192.168.1.100";
                id(ha_api_url) = "http://192.168.1.100:8123";
              } else if (ssid == "VanNetwork") {
                new_location = 2;
                id(mqtt_broker) = "192.168.2.100";
                id(ha_api_url) = "http://192.168.2.100:8123";
              } else if (ssid == "BoatNetwork") {
                new_location = 3;
                id(mqtt_broker) = "192.168.3.100";
                id(ha_api_url) = "http://192.168.3.100:8123";
              }
              
              if (new_location != id(current_location)) {
                id(current_location) = new_location;
                id(location_changed).execute(new_location);
              }

  - name: location_assist  # house_assist, van_assist, boat_assist
    stt_engine: stt.whisper
    tts_engine: tts.piper
    conversation_engine: homeassistant

  - platform: whisper
    model: tiny-int8  # Fast, accurate enough for commands
    language: en

tts:
  - platform: piper
    voice: en_US-lessac-low  # Small, natural-sounding

7. Display & UI Decisions

7.1 Screen Layout Options

Layout Best For Pros Cons
Tile Grid Many controls Familiar, quick access Can be cluttered
List View Scrolling data Clean, readable Slower navigation
Circular Menu Boat navigation Nautical aesthetic, one-handed Learning curve
Card Stack Mixed content Organized, swipeable Requires more gestures
Dashboard At-a-glance info Information dense Limited interaction
Decision: Use Card Stack for main navigation, Dashboard for boat instruments, Tile Grid for controls.

7.2 Always-On Display Strategy

Mode Trigger Content Power Impact
Full Active Touch, wrist wake, voice Full color, 60Hz High
Ambient Timeout (30s no interaction) Time, location, critical alerts Medium
Deep Sleep Button press, no motion None (screen off) Low
IMU Wake Wrist flick Full Active Configurable

IMU Configuration

sensor:
  - platform: bmi270  # or similar IMU
    id: imu
    on_motion:
      then:
        - lambda: |-
            // Detect wrist flick
            if (id(imu).get_accel_z() > 1.5) {
              id(display).set_brightness(100);
              id(lvgl).resume();
            }

9. Data Synchronization Strategy

9.1 State Management

Data Type Sync Direction Frequency Method
Sensor readings Watch → HA Real-time MQTT publish
Device states HA → Watch Real-time MQTT subscribe
Time/Location HA → Watch On connect API call
Configuration HA → Watch On change MQTT retained
Voice responses HA → Watch On demand Streaming

9.2 Offline Capability

Scenario Behavior Recovery
WiFi lost Show last known state, queue commands Auto-reconnect, sync on restore
HA unreachable Local control only (if applicable), show offline Retry with backoff
MQTT broker down Buffer locally, HTTP fallback Alert user, queue messages
Deep sleep wake Quick state refresh, minimal sync Resume normal operation

10. Security Considerations

Layer Threat Mitigation
WiFi Unauthorized access WPA3, strong passwords, MAC filtering
MQTT Eavesdropping, injection TLS encryption, client certificates
HTTP API Replay attacks, brute force HTTPS, API keys, rate limiting
Voice False wake, command spoofing Local wake word, confirmation tones
Physical Device theft, tampering Remote wipe (if cloud connected), encrypted storage
